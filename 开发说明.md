## 环境说明

当前默认工作环境是 docker 容器：

```bash
bash scripts/setup/run_container.sh
conda activate test-v1
```

后续所有 SAME 实验都默认在这个环境里执行。

## 开发原则

当一个功能涉及子项目时，按下面优先级处理：

1. 优先从父项目适配，例如新增脚本、配置文件、归档流程，不直接改 `third_party/` 源码。
2. 如果必须修改子项目代码，优先通过 `patches/` 管理，可参考 `patches/same/base/0001-eval-only-exit.patch` 和 `docs/patches/same.md`。
3. 如果 patch 方式明显不适合，请直接说明理由，我会自己考虑 fork 子项目；不要死硬地只用 patch，也不要擅自直接改 `third_party/` 工作树。

## SAME 实验入口

SAME v1 统一使用父项目入口：

```bash
# 主实验(目前使用test-v1环境,以后会修改这个问题)
bash scripts/setup/run_container.sh
conda activate test-v1
python scripts/experiments/run_same.py --config configs/same/val_r2r_eval_only.yaml
python scripts/experiments/run_same.py --config configs/same/val_r2r_reverie_cvdn_soon.yaml

# 细粒度分析
bash scripts/setup/run_container.sh
conda activate test-v1
python scripts/analysis/build_same_fine_metrics.py \
  --experiment-dir experiment_outputs/<experiment_id> \
  --connectivity-dir data/same/simulator/connectivity

# oracle gap / endpoint 上界报告
python scripts/analysis/build_oracle_gap_report.py \
  --experiment-dir experiment_outputs/<experiment_id>

# 逐样本分析
bash scripts/setup/run_mp3dsim_container.sh
python scripts/analysis/render_same_eval_item.py \
  <internal_item_id> \
  --experiment-dir experiment_outputs/<experiment_id> \
  --download-missing-scan

```

该入口会自动完成：

1. 应用 `patches/same/base/` 下的 base patches。
2. 生成规范化 `experiment_id`。
3. 从 `third_party/SAME/src` 启动 `run.py`。
4. 让 SAME 原始输出直接写入 `experiment_outputs/<experiment_id>/`。
5. 生成实验归档文件，并更新 `reports/tables/runs.csv` 与 `reports/tables/metrics_long.csv`。
6. 对 `reports/tables/official_results.csv` 做只读参考校验。

可选参数：

```bash
python scripts/experiments/run_same.py \
  --config configs/same/val_r2r_eval_only.yaml \
  --seed 0 \
  --checkpoint ../../../data/same/ckpt/SAME.pt \
  --tag smoke \
  --option training.workers=2
```

## experiment_id 规范

`experiment_id` 采用：

```text
NNNN_<method>_<config-stem>_<ckpt-tag>_s<seed>[_<tag>]_v<rev>
```

示例：

```text
0001_same_val_r2r_eval_only_same_s0_v1
0002_same_val_r2r_eval_only_same_s0_smoke_v1
0003_same_val_r2r_eval_only_same_s0_v2
```

规则：

1. `NNNN` 是全仓库单调递增的数字前缀，至少 4 位。
2. `method` 当前固定为 `same`。
3. `config-stem` 来自配置文件名，`ckpt-tag` 来自 checkpoint 文件名。
4. `s<seed>` 明确记录随机种子。
5. 同一语义 slug 重跑时，`v<rev>` 自动递增。
6. 只使用小写 ASCII；段之间用 `_`，段内允许 `-`。

## experiment_outputs

每次实验都要在 `experiment_outputs/<experiment_id>/` 下至少包含下面 9 个归档文件：

```text
run.json
config_resolved.yaml
metrics.json
stdout.log
stderr.log
git_info.txt
data_manifest.txt
gpu_info.txt
patch.diff
```

同时保留 SAME 原始产物，便于二次分析和人工复查：

```text
<experiment_id>.log
results/
ckpts/
tensorboard/
```

说明：

1. `run.json` 记录运行命令、状态、退出码、commit、patch_set、warning 等元数据。
2. `config_resolved.yaml` 是运行前合并后的最终配置快照。
3. `metrics.json` 是从 SAME 日志解析出的结构化 split-level 指标摘要。
4. `data_manifest.txt` 记录本次运行实际引用的数据、标注、特征和权重路径。

## NavNuances via SAME

NavNuances 先伪装成 SAME 的 R2R eval split 跑推理，再用 NavNuances evaluator 做细粒度评分。

`NavNuances` 只是和 `R2R` 风格相似,但是任务的各个样本无法对应,同时只有起点没有终点,评估指标也不同,因此在这种推理时,请忽视 `SAME` 实验代码给出的推理结果,而应该看`navnuances_eval`给出的推理结果.

```bash
# 将 navnuances 数据集伪装成 SAME 的 R2R eval 风格
python scripts/setup/prepare_navnuances_same_r2r.py

# 对 navnuances 数据集进行推理
bash scripts/setup/run_container.sh
python scripts/experiments/run_same.py \
  --config configs/same/navnuances_r2r_eval_only.yaml \
  --tag navnuances

# 评估 SAME 对 navnuances 的推理结果
bash scripts/setup/run_navnuances_container.sh
conda activate navnuances-eval

python scripts/eval/run_same_navnuances_eval.py \
  --experiment-id <experiment_id>
```

该入口默认写入：

```text
experiment_outputs/<experiment_id>/navnuances_submission/
experiment_outputs/<experiment_id>/navnuances_eval/
```

`prepare_navnuances_same_r2r.py` 默认会在存在 `data/navnuances/annotations/NavNuances/R2R_val_unseen.json` 时额外生成：

```text
data/navnuances/same/R2R/R2R_val_unseen_enc.json
```

`run_same_navnuances_eval.py` 会自动检测 `R2R_val_unseen_results.json` 与 `R2R_val_unseen.json`。两者都存在时同时评 Standard；否则只评 DC/LR/RR/VM/NU。

如果要确认 SAME 标准 split 与 NavNuances annotation_root 里的 Standard split 是否一致(应该是完全一致的)：

```bash
python scripts/analysis/compare_r2r_val_unseen.py
```

如果只想导出 submission，不跑 evaluator：

```bash
python scripts/eval/export_same_navnuances_submissions.py \
  --experiment-id <experiment_id>
```

## NavNuances via NavGPT-2

NavGPT-2 的适配原则和 SAME 一样：先把 NavNuances 五类 split 伪装成 R2R annotation 跑推理，再把 `submit_*.json` 交给 NavNuances evaluator 评分。

这里同样不要看 NavGPT-2 自带的 R2R validation 指标。NavNuances 只有起点，`R2R_*_enc.json` 里的 `path` 只是为了让 R2R 风格环境能 reset；最终指标必须以 `navnuances_eval/results.json` 为准。

### 当前接口判断

`research/VLN research/stage_traces/S.01.02.01.A1-NavGPT-2评估接口确认.md` 已确认：

1. NavGPT-2 XL 标准 R2R eval 已跑通，`BATCH_SIZE=1` 在当前 8GB 级显存环境可用。
2. NavGPT-2 读取的核心字段是 `scan/path_id/path/heading/instructions/instr_encodings`，NavNuances 转成 `R2R_*_enc.json` 后字段兼容。
3. 当前阻塞不是样本格式，而是 NavGPT-2 eval split 写死为 `val_train_seen/val_seen/val_unseen`。
4. NavGPT-2 默认输出 `preds/submit_<split>.json`，字段是 `instr_id/trajectory`，可直接作为 NavNuances submission 的主体。

### 已实现工程入口

按开发原则，优先在父项目新增脚本和 patch，不直接手改 `third_party/NavGPT-2` 工作树。

1. 先复用 `scripts/setup/prepare_navnuances_same_r2r.py` 生成 `data/navnuances/same/R2R/R2R_{DC,LR,RR,VM,NU}_enc.json`。
2. 再用 `scripts/setup/prepare_navnuances_navgpt2_r2r.py`，把这些 enc 文件复制或软链接到 NavGPT-2 数据根的 `R2R/annotations/` 下：

```text
data/navgpt2/datasets/R2R/annotations/R2R_DC_enc.json
data/navgpt2/datasets/R2R/annotations/R2R_LR_enc.json
data/navgpt2/datasets/R2R/annotations/R2R_RR_enc.json
data/navgpt2/datasets/R2R/annotations/R2R_VM_enc.json
data/navgpt2/datasets/R2R/annotations/R2R_NU_enc.json
```

`prepare_navnuances_navgpt2_r2r.py` 默认使用 `$NAVGPT2_DATASETS`，因此推荐在 `scripts/setup/run_navgpt2_container.sh` 进入的容器内运行；如果在宿主机运行，需要显式传 `--navgpt2-data-root "$NAVGPT2_HOST_ROOT/datasets"`。

3. 用 `patches/navgpt2/base/0002-eval-splits.patch` 给 NavGPT-2 增加 `--eval_splits` 参数。该 patch 属于 base 级别，会随 `scripts/setup/apply_patches.sh --method navgpt2` 一起自动应用。

推荐语义：

```bash
EXTRA_ARGS="--eval_splits DC LR RR VM NU"
```

4. 用 `scripts/eval/run_navgpt2_navnuances_eval.py` 做后处理：
   - 输入 `--experiment-dir experiment_outputs/<run_id>` 或 `--experiment-id <run_id>`。
   - 检查 `preds/submit_{DC,LR,RR,VM,NU}.json` 是否存在。
   - 将文件复制到 `experiment_outputs/<run_id>/navnuances_submission/`。
   - 调用 NavNuances evaluator，并把结果写到 `experiment_outputs/<run_id>/navnuances_eval/results.json`。

### 运行流程

NavGPT-2 推理在 NavGPT-2 容器里跑：

```bash
# 应用 NavGPT-2 base patch（包含 eval_splits patch）
bash scripts/setup/apply_patches.sh --method navgpt2

# 进入 NavGPT-2 推理环境
bash scripts/setup/run_navgpt2_container.sh
conda activate navgpt2

# 准备 NavNuances enc annotation，并安装到容器内的 NavGPT-2 数据根
python scripts/setup/prepare_navnuances_same_r2r.py
python scripts/setup/prepare_navnuances_navgpt2_r2r.py

RUN_ID="navgpt2_navnuances_xl_$(date +%Y%m%d_%H%M%S)"
OUTPUT_DIR="/workspace/vln-lab/experiment_outputs/${RUN_ID}" \
BATCH_SIZE=1 \
CUDA_VISIBLE_DEVICES=0 \
EXTRA_ARGS="--eval_splits DC LR RR VM NU" \
bash third_party/NavGPT-2/map_nav_src/scripts/val_r2r_xl.sh
```

NavNuances 评分在 NavNuances evaluator 容器里跑：

```bash
bash scripts/setup/run_navnuances_container.sh
conda activate navnuances-eval

python scripts/eval/run_navgpt2_navnuances_eval.py \
  --experiment-dir experiment_outputs/<run_id>
```

如果只想手动跑 evaluator，等价流程是：

```bash
mkdir -p experiment_outputs/<run_id>/navnuances_submission
cp experiment_outputs/<run_id>/preds/submit_{DC,LR,RR,VM,NU}.json \
  experiment_outputs/<run_id>/navnuances_submission/

python third_party/navnuances/evaluation/eval.py \
  --skip-standard \
  --annotation_root data/navnuances/annotations/NavNuances \
  --submission_root experiment_outputs/<run_id>/navnuances_submission \
  --out_root experiment_outputs/<run_id>/navnuances_eval \
  --connectivity_dir data/same/simulator/connectivity
```

最终用于 Ability Atlas 的 NavGPT-2 指标来自：

```text
experiment_outputs/<run_id>/navnuances_eval/results.json
```

## reports/tables

长期维护 3 张表：

```text
reports/tables/official_results.csv
reports/tables/runs.csv
reports/tables/metrics_long.csv
```

### official_results.csv

用途：人工维护官方参考结果。运行器只读，不自动写入。

表头：

```text
source,method,dataset,split,metric,value,note
```

### runs.csv

用途：记录每次实验到底跑了什么、在哪个版本上跑的。
其中 `duration_hms` 使用 `HH:MM:SS` 格式，例如 `00:04:55`。

表头：

```text
experiment_id,date,run_type,method,datasets,splits,repo_commit,child_repo_commit,config,checkpoint,seed,status,duration_hms,log_path,output_dir,patch_set
```

### metrics_long.csv

用途：把每次实验的指标摊平成长表，便于后续筛选、透视和画图。

表头：

```text
experiment_id,dataset,split,metric,value,unit
```

约定：

1. 百分比指标统一写 `%`。
2. 距离类指标统一写 `m`。
3. 计数或无量纲指标的 `unit` 留空。
