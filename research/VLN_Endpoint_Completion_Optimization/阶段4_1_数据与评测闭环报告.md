## 1. 本阶段作用

阶段 4.1 对应总路线中“离线 endpoint learning 闭环”的第一步：

```text
如何把 SAME trace 固定成可训练、可评测的 endpoint candidate group？
```

它的作用不是训练 gate，也不是训练 ranker，而是先把后续 learned method 必须共享的数据协议和评测出口固定下来：

```text
episode group：一个 episode 对应一个 endpoint candidate group
candidate：trajectory 中的每个 step 都是一个候选 endpoint
label：每个 candidate 是否成功、SPL、reward、nearest / best-success 标记
pair：由 candidate label 构造 preference pair
gate label：should_rerank = final_success == false AND oracle_success == true
evaluator：任意 gate / ranker 只要输出统一 score CSV，就能生成 tau curve 和 recovery-harm curve
```

对于总路线来说，阶段 4.1 的作用是把阶段 4 后续工作从“写一个模型试试看”变成严格离线闭环：

```text
4.2 gate-only baseline：只需要输出 gate_score / should_rerank_score
4.3 endpoint ranker baseline：只需要输出 candidate_score
4.4 frozen config：只在 train/dev 上选择 gate_threshold、tau、allow_change_final
4.6 val_unseen final report：只用冻结配置一次性报告，不反向调参
```

本阶段只回答“数据和 evaluator 是否已经统一”。`stop_prob_scores.csv` 只是 smoke score，用来验证 evaluator 出口可运行，不作为 learned method 或阶段 4 的方法结论。

## 3. 代码实现注意事项

### 3.1 数据对齐

脚本复用阶段 1 / 2 的数据发现与 fine metrics 对齐方式：

```text
eval_items/*_eval_context.json
eval_items/*_eval_items.jsonl
fine_metrics/tables/fine_metrics_wide.csv
```

每条 item 通过下面的键和 `fine_metrics_wide.csv` 对齐：

```text
dataset
split
internal_item_id
```

如果 fine metrics 中缺少对应行，脚本会直接报错，而不是静默跳过。

### 3.2 candidate 协议

当前固定协议是：

```text
one episode = one group
one trajectory step = one candidate
不做 viewpoint deduplication
candidate = all trajectory steps
target_scope = official
```

这意味着 repeated viewpoint / loop / route-expanded intermediate step 都会保留为候选 endpoint。后续 ranker 需要自己学习“哪个 step 更值得停”，而不是由数据构造阶段提前删掉候选。

### 3.3 trace 对齐

SAME 的 `decision_trace.steps` 与 `prediction.trajectory` 不一定是一一对应关系。脚本通过 `current_viewpoint` 和 `route_viewpoints` 对齐：

```text
有 decision trace 的 step：保留 stop_prob、selected_prob、router entropy 等特征
route-expanded intermediate step：保留 label，但 has_decision_trace=false
无法对齐的 trace step：不强行写入 candidate
```

这点很重要：candidate label 以 trajectory step 为准，模型可用特征以可对齐的 SAME decision trace 为准。

### 3.4 train / dev / test 协议

当前协议：

```text
val_train_seen：stable hash split，dev_ratio = 0.2
val_unseen：test/report only
dev_salt：endpoint_learning.v1
```

也就是说：

```text
train：用于训练 gate / ranker
dev：用于选择 gate_threshold、tau、allow_change_final
test：对应 val_unseen，只能用于冻结配置后的最终报告
```

### 3.5 label 与 feature 边界

candidate 可用于训练 / 推理的 trace features 包括：

```text
is_final
has_decision_trace
step_frac
path_length_m
stop_prob
stop_margin_prob
selected_prob
top1_top2_margin
moe_router_entropy
fuse_weight
is_revisit
is_loop_region
is_last_k
```

下面字段只能用于离线 label、loss 或 evaluation，不能作为 gate / ranker 的推理输入：

```text
success_label
spl_at_candidate
distance_to_goal_m
reward
is_best_success_candidate
is_nearest_candidate
final_success
oracle_success
nearest_endpoint_success
should_rerank
```

其中 `distance_to_goal_m` 是 GT distance，必须严禁进入 inference feature。

### 3.6 pair 构造

当前构造三类 preference pair：

```text
success_gt_fail：成功 candidate > 失败 candidate
better_spl_success_gt_lower_spl_success：高 SPL 成功 candidate > 低 SPL 成功 candidate
final_success_final_gt_failed_nonfinal：final 已成功时，final > 失败 non-final
```

第三类 pair 是阶段 3 失败分析之后加入的保守约束：它服务于“不要轻易伤害原本成功的 final endpoint”。

### 3.7 evaluator 协议

evaluator 不关心 score 来自规则、gate、ranker 还是 ablation，只要求输入 score CSV 至少包含：

```text
candidate_id
candidate_score
```

可选包含：

```text
gate_score
```

当前推理规则：

```text
if gate_score < gate_threshold:
    choose final
else:
    choose argmax candidate_score
    if best_score <= final_score + tau:
        choose final
```

输出指标统一为：

```text
SR / SPL / delta_SR / delta_SPL
recovery_rate / harm_rate / net_recovery_rate
changed_endpoint_rate
gate_precision / gate_recall / gate_auc
tau_curve
recovery_harm_curve
```

这说明阶段 4.1 的 gate label 与阶段 1 / 2 的 final failure / oracle success 口径是对齐的。

### 4.4 与阶段 1 / 2 / 3 的衔接

阶段 1 给出的核心事实是：

```text
val_train_seen oracle gap = 84
val_unseen oracle gap = 200
```

阶段 4.1 中：

```text
val_train_seen train should_rerank = 68
val_train_seen dev should_rerank = 16
val_unseen test should_rerank = 200
```

因此 4.1 的 `should_rerank` 标签已经正确继承了阶段 1 的 oracle gap 口径。

阶段 2 证明 nearest endpoint 是 upper bound；阶段 4.1 保留：

```text
nearest_endpoint_success
nearest_endpoint_spl
is_nearest_candidate
```

这些字段只能用于 evaluation / label analysis，不能进入 inference feature。

阶段 3 证明 rule-only reranker 的主要风险是 harm 原本成功的 final endpoint；阶段 4.1 因此加入：

```text
final_success_final_gt_failed_nonfinal pair
allow_change_final
tau
harm_rate
final_success_harm_rate
```

这些字段和指标为 4.2 / 4.3 的 conservative gate + ranker 提供了统一评测出口。
