# 阶段4 总结与下一步计划

状态：阶段 4 收束总结，待人工审核

本文档是阶段 4 的路线级收束文档，只记录阶段结论、关键证据链、止损决定和下一步计划。具体实验命令、完整表格和产物路径仍以 `阶段4_1_数据与评测闭环报告.md` 到 `阶段4_7_dev 止损与 val_unseen 一次性报告.md` 为准。

## 1. 总结论

阶段 4 已经足够支持一个明确判断：

```text
endpoint completion 研究问题成立；
但当前 single-trajectory trace-only gate + ranker 框架不应继续扫参数、loss、tau 或 threshold。
```

更具体地说：

| 问题 | 当前判断 |
| --- | --- |
| endpoint gap 是否真实存在 | 存在，且上界较大 |
| SAME trace 是否有 endpoint correction 信号 | 有弱信号，但 trace-only 不稳定 |
| CE ranker 是否足够 | 不足，candidate-level success 与 endpoint top1 preference 错位 |
| preference objective 是否有价值 | 有价值，尤其 final-bias 解耦证明 gate/ranker 职责需要分离 |
| 4.7 `val_unseen` 是否支持继续当前框架 | 不支持，SR recovery 基本被 harm 抵消 |
| 阶段 4 是否进入 NavNuances router 主实验 | 不进入，当前框架收束为 weak / negative result |

阶段 4 的稳妥表述是：

```text
阶段 4 证明了 endpoint oracle gap 与 trace signal 的存在，
也证明了 CE ranker 目标错位和 final-bias 解耦的必要性；
但 frozen val_unseen 结果显示，当前 trace-only gate + ranker
无法稳定把 dev recovery 外推到 unseen split，
SR recovery 被 harm 基本抵消。
```

因此，当前实现止损；endpoint completion 方向保留，但下一步必须引入新证据源、gate 泛化诊断或多轨迹共识。

## 2. 关键证据链

### 2.1 阶段 1-3：问题成立，但 rule-only 不够

阶段 1/2 证明 SAME 存在大量 final STOP / endpoint 可恢复失败。`val_unseen` 上，如果 oracle-style 在 SAME 已访问 endpoint 中选择最近目标的节点，weighted oracle gap 约为 13.49pp；其中 SOON 和 CVDN 的上界尤其大。

阶段 3 证明 STOP prob、loop、revisit、last-high-stop 等 trace 特征确实有恢复信号；但固定 rule 一旦更激进，harm 很快抵消 recovery。因此进入 learned gate + ranker 是合理的，当时方向没有错。

### 2.2 阶段 4.1-4.4：learned 闭环成立，但 CE baseline 太弱

4.1 固定了 candidate group、label、pair 和 evaluator 协议，建立了离线 endpoint learning 闭环。

4.2 证明 episode-level gate 可学，但 gate 主要解决“少改”，不能单独解决“改对”。

4.3/4.4 的 CE success ranker 可以学到 candidate-level success 信号，但不能稳定转化为 group top1 endpoint selection。4.4 frozen 配置在 joint dev 上 `delta_SR = +0.09pp`，`delta_SPL = +0.21pp`，净增只有 7 个成功 episode；它是 low-harm weak baseline，不是最终方法。

### 2.3 阶段 4.5：目标错位被定位

4.5 诊断说明：

```text
candidate-level success classification 不等于 episode-level endpoint preference。
```

CE ranker 对 success/fail 区分较强，但在同一条 trajectory 内的成功 endpoint 之间排序很弱，并表现出 late/final-like bias。它学到的是“像不像成功点”，不是“当前 episode 里哪个 endpoint 最值得停”。

### 2.4 阶段 4.6：preference 有效，但需要职责解耦

SPL-aware pairwise / group-listwise objective 能改善 success-success preference，但主 ablation 没有形成足够 endpoint recovery。

更有价值的是 final-bias 解耦 sanity：

```text
gate 管“要不要改 final”；
ranker 管“如果允许修改，选哪个 endpoint”。
```

该 sanity 在 dev 上明显改善 ranker top1 与 dSPL，说明 preference objective 方向不是完全无效；但它仍需要 gate 稳定判断何时放行。

### 2.5 阶段 4.7：最终断点在 val_unseen 泛化

4.7 frozen `val_unseen` 给出了阶段 4 的止损依据。

| 配置 | weighted dSR | weighted dSPL | recovery | harm | changed | 结论 |
| --- | ---: | ---: | ---: | ---: | ---: | --- |
| `safety` | 约 0.000pp | +0.064pp | 21 items | 21 items | 1.14% | 主配置 SR 打平 |
| `dev_best` | +0.039pp | +0.373pp | 87 items | 83 items | 4.79% | sensitivity 仅净增约 4 items，且单数据集 harm 不稳 |

`dev_best` 只能说明 ranker 能找到一部分更短或更好的 endpoint，不能包装成强方法结果。它对 oracle gap 的 net capture 约 0.29%，远低于可作为主要方法贡献的水平。

## 3. 为什么当前框架不理想

当前结果不是单点失败，而是几个瓶颈叠加：

1. 阶段 2 的 nearest endpoint 上界依赖 GT distance；真实 inference 只有 SAME trace 内部信号，缺少语义验证。
2. endpoint correction 的代价结构很尖锐；改少了没有收益，改多了 harm 快速上升。
3. ranker 的局部进步无法弥补 gate 的跨 split 泛化不足。
4. CVDN / SOON 更容易放大 trace-only false positive，统一 gate/ranker 覆盖四数据集风险较高。

因此继续扫下面这些项的边际价值很低：

```text
gate_threshold
tau
CE / pairwise / listwise loss weight
final bonus
hidden dim
epoch
```

这些更可能只是在 recovery-harm trade-off 曲线上移动，而不是创造新的可分性。

## 4. 阶段 4 冻结决定

本阶段冻结以下决定：

| 决定 | 说明 |
| --- | --- |
| 不继续当前 trace-only gate + ranker 参数扫 | 4.7 已完成 frozen val_unseen，一次性报告原则已执行 |
| 不用 `val_unseen` 反向选择 threshold / tau / objective | 维护协议冻结纪律 |
| 不把 `dev_best` 当主结果 | SR 只净增约 4 items，SOON / CVDN harm 不稳 |
| 暂停 NavNuances router 主实验 | 阶段 4 没有达到路线最低成功标准 |
| 暂不启动 full RL / GRPO | 当前瓶颈是证据不足与 gate 泛化，不是缺更复杂优化器 |

允许保留一个轻量方向：

```text
failure category / gate 泛化诊断。
```

它只用于解释 dev 到 val_unseen 的断点，并为下一阶段新证据设计服务，不作为新的 router 主实验。

## 5. 下一步计划

### 5.1 P0：gate 泛化诊断

新增一个只分析、不训练新模型的诊断阶段，建议命名为：

```text
阶段4_8_gate泛化诊断.md
```

最小产物包括四张表：

| 表 | 内容 | 目的 |
| --- | --- | --- |
| A | gate calibration by split / dataset | 看 dev 与 `val_unseen` 是否校准漂移 |
| B | gate threshold confusion | 判断是否存在真正分开 recovery/harm 的阈值 |
| C | changed item feature shift | 比较 recovered、harmed、changed-safe、changed-unrecovered 的轨迹特征 |
| D | dev vs `val_unseen` top1 ranker stability | 判断断点主要来自 gate 还是 ranker top1 漂移 |

该阶段不改变 4.7 frozen 结论，只解释为什么当前框架不能稳定外推。

### 5.2 P1：选择一个新方向，不同时开多个

优先推荐方向 A：

```text
evidence-augmented endpoint verification。
```

核心思想是：trace-only ranker 只产生 top-k endpoint；新增 evidence verifier 判断候选 endpoint 是否真的满足 instruction / object / dialog goal；gate 再根据 trace + verifier margin 决定是否修改 final。

最小版本可以先避免大规模 VLM，优先使用轻量证据：

```text
instruction keyword/entity overlap
room/object/landmark proxy
candidate visual/semantic metadata
trajectory prefix 与 instruction clause 对齐
dialog evidence for CVDN
```

备选方向 B：

```text
multi-trajectory / multi-sampling consensus endpoint completion。
```

即从单条轨迹证据改为多轨迹共识，只在多个 trajectory 都支持的 endpoint cluster 上出手。

备选方向 C：

```text
SPL-preserving / efficiency rerank。
```

该方向只在 SPL 收益能稳定放大、且 SR harm 可控时考虑。当前 4.7 的 SPL 正收益还太小，不能单独支撑主方法。

### 5.3 下一阶段最低继续线

如果新方向进入 `val_unseen`，建议至少满足：

| 指标 | 继续条件 |
| --- | ---: |
| `val_unseen` weighted dSR | >= +0.20pp |
| weighted harm | <= 1pp |
| max dataset harm | <= 1pp |
| CVDN / SOON dSR | 不明显为负 |
| net recovered items | >= 20 |
| gap capture | >= 1.5% 起步，>= 5% 才像样 |

当前 `dev_best` 只有净 +4 items，gap capture 约 0.29%；下一阶段必须明显超过这个基线，才值得继续扩展。

## 6. 可写入论文/报告的口径

这条线更适合写成 diagnostic / negative result，而不是包装成一个强 reranker：

```text
Modern VLN agents have a large endpoint oracle gap,
but this gap is not automatically recoverable from single-trajectory trace evidence.
```

中文口径：

```text
SAME 的 endpoint oracle gap 很大；
但只依赖单条轨迹内部 trace，很难安全恢复。
```

可作为贡献的证据链：

1. SAME 已访问 endpoint 中存在大量 oracle-recoverable failure。
2. Rule-only trace 信号存在，但 harm 不可控。
3. CE ranker 的 candidate-level success 与 episode-level endpoint preference 错位。
4. Preference objective 能改善一部分排序问题。
5. Final-bias 解耦说明 gate/ranker 职责必须分离。
6. Frozen `val_unseen` 显示 trace-only gate 泛化不足，recovery 被 harm 抵消。

最终建议：

```text
阶段 4 当前框架止损；
endpoint completion 方向保留；
下一步转向 gate 泛化诊断、证据增强 endpoint verification 或多轨迹共识。
```
