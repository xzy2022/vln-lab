# 研究记录规范

本规范基于三层地图结构（global / local / stage）组织研究记录，每一层文件内部遵循统一的认知工作流。

## 一、目录结构

```
research/VLN research/
├── global_maps/   # 全局地图，代表大的研究方向。命名为 `NN.<name>.md`
├── local_maps/    # 局部地图，将大的研究方向拆成多个阶段。命名为 `NN.NN.<name>.md`
└── stage_maps/    # 阶段地图，记录阶段内的每一次决策。命名为 `NN.NN.NN.<name>.md`
```

各层的编号规则：
- `global_maps/`：`NN.<name>.md`，如 `01.端点补全优化.md`
- `local_maps/`：`NN.NN.<name>.md`，如 `01.02.gate泛化诊断.md`
- `stage_maps/`：`NN.NN.NN.<name>.md`，如 `01.02.03.gate泛化诊断_0428.md`

## 二、文件内容结构

每一层地图的文档都应包含以下模块（按顺序组织），各模块对应认知工作流的一个环节：

| 模块 | 工作流层级 | 作用 |
|------|-----------|------|
| **目标 / 任务** | 定义层 | 当前要解决什么问题，成功标准是什么 |
| **背景与输入** | 输入层 | 外部事实（论文、官方文档）和内部事实（实验结果、日志、观察） |
| **共识与经验** | 输入层 | 学术共识、工程经验、个人历史经验中高压缩的规律 |
| **理解与解释** | 认知层 | 对事实和共识做初步诠释 |
| **问题建模** | 认知层 | 将问题从模糊状态变为结构化对象：系统边界、关键变量、因果链条、失败位置、指标关系、约束条件 |
| **假设** | 认知层 | 在问题模型中提出可检验主张：核心假设、支撑假设、隐含假设 |
| **验证设计** | 验证层 | 设计能够区分不同假设的动作与判别标准 |
| **当前判断** | 控制层 | 根据已有材料，定位当前状态 |
| **决策分析** | 控制层 | 提出可行路线，分析成本收益，并做取舍 |
| **行动计划** | 执行层 | 将决策转化为步骤、资源安排（时间、算力）、优先级、停止条件 |
| **执行记录** | 执行层 | 按计划实施，产出代码、实验、文档或其他结果（实时更新） |
| **结果复盘** | 反思层 | 回看目标、假设、验证、判断、决策与结果之间是否一致，提炼经验 |

## 三、命名规范

### 文件命名

文件名应包含足够的信息让人快速判断文件用途：
- 阶段名 + 关键变量或假设关键词（如 `gate泛化诊断`、`endpoint-bias解耦`）
- 日期后缀用于区分同一阶段内的多次决策记录（如 `_0428`、`_0501`）
- 避免纯日期或无意义代号

### 内部标题

每个模块使用二级标题（`##`），模块名称与上表一致。模块顺序不可随意打乱，但根据阶段实际需要可跳过不适用的模块。

## 四、写作要求

- **事实与观点分离**：背景与输入中只写事实，理解和解释中写观点。事实用引用标注来源（论文名、实验 ID、文件名）。
- **假设必须可证伪**：每个假设应能设计出一个验证动作和一个判别标准。
- **行动计划有停止条件**：每个计划明确在什么条件下停止或回退。
- **执行记录实时更新**：在实验运行期间持续更新，而非事后补写。
- **复盘模块必须存在**：每个 stage_map 文档在结束后必须有复盘，无进展也需要记录"未达预期"。

## 五、各层地图职责

| 层级 | 粒度 | 记录什么 | 何时更新 |
|------|------|----------|----------|
| global_map | 研究方向整体 | 方向动机、长期目标、整体战略选择 | 启动新方向时新建；战略方向变化时更新 |
| local_map | 方向内的阶段 | 阶段目标、关键假设、阶段边界、跨子阶段经验 | 进入新阶段时新建；重大发现或方向调整时更新 |
| stage_map | 阶段内单次决策 | 问题建模、单次假设、单次验证设计与结果 | 每轮实验迭代新建；结果出来后更新复盘 |

- SAME baseline val_unseen SR: 47.2%（官方参考）
- 本次实验 SR: 31.2%，下降 16pp
- 日志显示 early-stop 触发比例异常（见 stderr.log）
- 现有文献中关于 encoder-gate 解耦的讨论（[Hambucker et al. 2024]）

## 共识与经验
- gate 模块在 unseen 场景下更敏感（Hambucker 2024）
- endpoint bias 与 encoder 存在耦合（阶段 4.7 的消融实验）

## 理解与解释
- 从日志看，gate 在 val_unseen 上倾向于保守输出，导致 success rate 下降
- ...

## 问题建模
- 关键变量：gate_output_entropy、endpoint_distribution、encoder_hidden_std
- 因果链条：分布偏差 → gate 误判 → 轨迹偏离
- 失败位置：action selection 阶段

## 假设
1. **核心假设**：endpoint bias 泄漏到 gate 是泛化崩溃主因
2. **支撑假设**：移除 final-bias 解耦后，gate 泛化有所改善（待验证）

## 验证设计
- 实验 A：`--final-bias-decouple=false`（当前 baseline）
- 实验 B：`--final-bias-decouple=true` + 不同 lr
- 判别标准：val_unseen SR 恢复 ≥ 40%

## 当前判断
当前处于诊断初期，需要先运行实验 B 确认假设。

## 决策分析
- 路线 1（推荐）：先验证 final-bias 解耦假设，同时跑 gate-specific 分析脚本

（运行期间持续更新）

（实验结束后填写）
