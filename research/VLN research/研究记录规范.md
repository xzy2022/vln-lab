# 研究记录规范

本规范基于三层地图结构（global / local / stage）组织研究记录，每一层文件内部遵循统一的认知工作流。

## 一、目录结构

```
research/VLN research/
├── global_traces/    # 全局轨迹。命名 `G.NN.<name>.md`，`G` = Global，`NN` 为顺序编号
├── local_traces/     # 局部轨迹。命名 `L.NN.NN.<name>.md`，`L` = Local，第一段对应 global，第二段为阶段顺序
└── stage_traces/     # 阶段轨迹。命名 `S.NN.NN.NN.<name>.md`，`S` = Stage，三段分别对应 global / local / 执行轮次
```

各层的编号规则：
- `global_traces/`：`G.NN.<name>.md`，如 `G.01.端点补全优化.md`
- `local_traces/`：`L.NN.NN.<name>.md`，如 `L.01.02.gate泛化诊断.md`
- `stage_traces/`：`S.NN.NN.NN.<name>.md`，如 `S.01.02.03.gate泛化诊断_0428.md`

## 二、文件内容结构

每一层地图的文档包含的模块如下所示：


| 模块          | 工作流层级 | 所属层级           | 作用                                            |
| ----------- | ----- | -------------- | --------------------------------------------- |
| **目标 / 任务** | 定义层   | `global`       | 当前要解决什么问题，成功标准是什么                             |
| **背景与输入**   | 输入层   | `global`       | 外部事实（论文、官方文档）和内部事实（实验结果、日志、观察）                |
| **共识与经验**   | 输入层   | `global`       | 学术共识、工程经验、个人历史经验中高压缩的规律                       |
| **理解与解释**   | 认知层   | `global`       | 对事实和共识做初步诠释                                   |
| **问题建模**    | 认知层   | `global`       | 将问题从模糊状态变为结构化对象：系统边界、关键变量、因果链条、失败位置、指标关系、约束条件 |
| **假设**      | 认知层   | `global`       | 在问题模型中提出可检验主张：核心假设、支撑假设、隐含假设                  |
| **当前判断**    | 控制层   | `global`       | 根据已有材料，定位当前状态                                 |
| **决策分析**    | 控制层   | `global`       | 针对当前判断, 提供多个不同的路线方案, 供人类最终决策                  |
| **验证设计**    | 验证层   | `local`        | 根据人类选择的路线方案, 进行验证的思路设计                        |
| **行动计划**    | 执行层   | `local`        | 将验证思路转化为明确的可执行步骤, 供之后 AI 逐条实现.                |
| **执行记录**    | 执行层   | `stage`        | 人类根据 AI 逐步骤实现的效果, 予以反馈.                       |
| **结果复盘**    | 反思层   | `global/local` | 回看目标、假设、验证、判断、决策与结果之间是否一致，提炼经验                |


## 三、命名规范

### 文件命名

文件名应包含足够的信息让人快速判断文件用途：
- 阶段名 + 关键变量或假设关键词（如 `gate泛化诊断`、`endpoint-bias解耦`）
- 日期后缀用于区分同一阶段内的多次决策记录（如 `_0428`、`_0501`）
- 避免纯日期或无意义代号

### 内部标题

每个模块使用二级标题（`##`），模块名称与上表一致。模块顺序不可随意打乱，但根据阶段实际需要可跳过不适用的模块。

## 四、写作要求

- **事实与观点分离**：背景与输入中只写事实，理解和解释中写观点。事实用引用标注来源（论文名、实验 ID、文件名）。
- **假设必须可证伪**：每个假设应能设计出一个验证动作和一个判别标准。
- **行动计划有停止条件**：每个计划明确在什么条件下停止或回退。
- **执行记录实时更新**：在实验运行期间持续更新，而非事后补写。
- **复盘模块必须存在**：每个 stage_trace 文档在结束后必须有复盘，无进展也需要记录"未达预期"。
- **事实类用表格**：外部文献事实、个人实验事实、工程约束等统一用 Markdown 表格组织，便于引用和更新。
- **模块可嵌套子节**：背景与输入、共识与经验等模块可根据实际需要拆分二级子节（如 `### 外部文献事实`、`### 个人实验事实`），不强制所有子节都出现。

## 五、各层轨迹职责

| 层级 | 粒度 | 记录什么 | 何时更新 |
|------|------|----------|----------|
| global_trace | 研究方向整体 | 方向动机、长期目标、整体战略选择 | 启动新方向时新建；战略方向变化时更新 |
| local_trace | 方向内的阶段 | 阶段目标、关键假设、阶段边界、跨子阶段经验 | 进入新阶段时新建；重大发现或方向调整时更新 |
| stage_trace | 阶段内单次决策 | 问题建模、单次假设、单次验证设计与结果 | 每轮实验迭代新建；结果出来后更新复盘 |

## 六、各层之间的数据流动

global_trace 向下游输出：
- **方向动机**：为什么做这件事，作为 local_trace 的背景
- **战略选择**：放弃什么、坚持什么，作为 stage_trace 决策分析的约束边界
- **阶段路线图**：global_trace 中的 Stage A/B/C/... 作为 local_trace 的阶段边界参考
- **贡献设计**：C1/C2/... 的贡献体系指导 stage_trace 的验证设计优先级

local_trace 向 stage_trace 输出：
- **阶段假设**：local_trace 中提炼的核心/支撑假设，直接作为 stage_trace 的输入
- **阶段边界**：local_trace 定义的阶段开始/结束标志，是 stage_trace 的成功标准参照
- **跨阶段经验**：前一个 stage 的复盘结论，影响下一个 stage 的假设修正

stage_trace 向上游反馈：
- **单次实验结论**：更新 local_trace 的假设检验状态
- **Negative diagnostic**：无效假设记录在 local_trace 中，避免重复
- **贡献素材**：stage_trace 的结果直接支撑 global_trace 的贡献体系

```plaintext
global_trace (方向战略)
    ├── 提供: 方向动机、战略选择、阶段路线图、贡献体系
    │
    ▼
local_trace (阶段假设)
    ├── 细化: 将 global 假设拆为阶段级核心/支撑假设
    ├── 定义: 阶段开始/结束的成功标准
    ├── 积累: 跨 stage 的经验教训
    │
    ▼
stage_trace (单次验证)
    ├── 执行: 针对单个假设的验证设计 + 行动计划
    ├── 记录: 执行过程与结果
    └── 复盘: 假设是否被证实/证伪，输出到 local_trace
```

### Global 示例 G.01.xxx.md
```markdown

## 目标 / 任务

学术性或理论性或任务性或验证性的目标或任务,而非一个在什么什么数据集上的性能提高多少多少.

## 背景与输入

### 外部文献事实

| 论文 | 核心贡献 | 对本诊断的意义 |
|---|---|---|
| Hambucker et al. 2024 | encoder-gate 解耦讨论 | gate 模块在 unseen 场景下更敏感的理论依据 |
| ... | ... | ... |

### 个人实验事实

- SAME baseline val_unseen SR: 47.2%（官方参考）
- 本次实验 SR: 31.2%，下降 16pp
- 日志显示 early-stop 触发比例异常（见 stderr.log）

### 工程约束

| 约束 | 对诊断的影响 |
|---|---|
| 研一学生，算力有限 | 优先诊断性脚本，避免大规模扫参 |
| val_unseen 只能一次性报告 | 诊断在 dev/val_seen 上完成 |

## 共识与经验

- gate 模块在 unseen 场景下更敏感（Hambucker 2024）
- endpoint bias 与 encoder 存在耦合（阶段 4.7 的消融实验）
- 任何 endpoint correction 方法要先看 recovery/harm，不能只看平均 SR/SPL

## 理解与解释

从日志看，gate 在 val_unseen 上倾向于保守输出，导致 success rate 下降。
...（更多解释）

## 问题建模

### 系统边界

```
输入：SAME backbone 输出（action distribution、trajectory、candidate endpoints）
模块：gate decision layer
输出：stop / continue binary decision
```

### 关键变量

| 变量 | 类型 | 说明 |
|---|---|---|
| `gate_output_entropy` | continuous | gate 输出的熵，泛化崩溃时偏高 |
| `endpoint_distribution` | categorical | final / candidate endpoint 的分布偏差 |
| `encoder_hidden_std` | continuous | encoder 隐状态的方差 |

### 因果链条

分布偏差 → gate 误判 → 轨迹偏离 → SR 下降

### 失败位置

action selection 阶段（第 12 步附近的 stop decision）

## 假设

必须区分`核心假设`,`支撑假设`,`隐含假设`。一般优先验证核心假设，除非某个支撑假设很关键且容易验证。



## 当前判断
1. 当前的目标是什么(我全部的假设和研究到底是要为了解决什么任务)
2. 为了实现这个目标,当前面临什么困难和问题
3. 关键假设的证明能否解决这些困难或问题?
4. 关键假设的证伪能否转换这些困难或问题?
5. 如果3和4的回答都是否,意味着什么?我此刻还能做些什么?

## 决策分析
针对当前判断,提供多个不同的路线方案,对于每条路线分别分析:
1. 该路线的直接目的是什么,二选一(A. 搜集更多信息,以为验证某个假设做准备; B. 信息已经足够,尝试验证某个假设)
2. 路线的主要思路是什么?
3. 该路线完成时可能会出现哪些结果?分别意味着什么?该如何面对?
4. 可行性与风险评估(时间,算力等资源,实施难度等等,我只有自己一个人+AI工具辅助)
严禁只给出一个路线方案,然后分析不同的阶段.这是在确定路线方案之后,`行动计划`部分做的事.
```

### 写作风格要求

- **事实类用表格**：外部文献事实、个人实验事实、工程约束、工程经验统一用 Markdown 表格组织，便于引用和更新
- **模块可嵌套子节**：背景与输入、工程约束、共识与经验等模块可根据实际需要拆分二级子节，不强求所有子节都出现
- **TBD 在第一次生成时留空**：验证设计、行动计划在AI第一次生成时留空,只有人类根据`决策分析`结果明确选定路线,且发出指令后,才根据路线在局部层级生成.
- **复盘不能空白**：每个 stage_trace 文档在结束后必须有复盘，即使"未达预期"也要写明原因
