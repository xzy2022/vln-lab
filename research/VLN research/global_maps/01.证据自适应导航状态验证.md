# 01.证据自适应导航状态验证

## 目标 / 任务

将研究主线从 STOP endpoint 修补升级为**证据自适应导航状态验证（Evidence-Adaptive Navigation State Verification, EANSV）**框架。

核心问题：训练型 VLN agent 在不确定状态下，如何通过额外证据安全地验证当前状态、候选 endpoint、instruction progress 与目标语义。

成功标准：
- offline verifier 的 recovered/harmed evidence margin 能在 dev 与 val_seen 上显著分开
- weighted val_unseen dSR >= +0.20pp，harm <= 1pp，max dataset harm <= 1pp
- verifier 调用率在高 uncertainty 下显著高于低 uncertainty
- NavNuances 五大能力切片至少有 DC / NU / LR / RR / VM 的诊断解释

## 背景与输入

### 外部文献事实

| NavNuances (EMNLP Findings 2024) | 五类细粒度能力评估 | 将 R2R aggregate 转向 ability-slice 诊断 |
| AdaNav (arXiv 2025) | uncertainty-based adaptive reasoning | uncertainty-triggered reasoning 的直接参考 |

### 个人实验事实

- SAME 已完成复现，R2R val_unseen SR=76.29, SPL=66.24；REVERIE val_unseen SR=45.84；CVDN/SOON val_unseen 已有基线数据
- NavNuances 复现揭示 SAME 与 NavGPT 互补：SAME 强 RR/VM（89.82/85.88），NavGPT 强 DC/LR（92.68/62.87），NU 普遍弱（~30-40%）
- Endpoint Completion 旧路线已证明：single-trajectory trace-only gate + ranker 无法稳定恢复 endpoint gap，stage 4.7/4.8 显示 recovery ≈ harm
- SAME checkpoint eval 已成为可复用工程资产

## 共识与经验

### 学术共识

1. VLN 从单任务特化转向统一训练/多任务 agent（SAME 为代表）
2. 从 aggregate benchmark 转向 fine-grained diagnosis（NavNuances 为代表）
3. 从 LLM 直接导航转向 LLM 作为 reasoning/copilot/verifier（NavGPT-2、VLN-Copilot、AdaNav）
4. 从固定推理转向 uncertainty-adaptive reasoning（AdaNav）
5. 从 final outcome reward 转向 step-level progress / long-horizon consistency（NavBench、LH-VLN、SACA）

### 工程经验

1. 对当前资源，复现与扩展 SAME 比训练新 backbone 更稳
2. 轻量 offline verifier、数据诊断、evidence extraction 性价比高于直接 full training
3. val_unseen 只能做冻结后一次性报告；阈值/参数选择须在 train/dev 或 val_seen 上完成
4. 任何 endpoint correction 方法要先看 recovery/harm，不能只看平均 SR/SPL
5. LLM/MLLM 调用要有 budget 与 trigger，否则成本高且难复现

### 个人经验

- 已亲自验证：单条轨迹内部 trace 对 STOP/endpoint 有弱信号，但不足以形成稳定方法
- 已有 negative diagnostic 可直接作为新路线的动机：trace-only insufficient → 需要外部证据

## 理解与解释

### 为什么 LLM 不能直接替代训练型 backbone

NavGPT 的优势在 reasoning 与语言理解，但离散 VLN 的成功需要长期 trajectory consistency、topological state、action grounding、stop calibration、visual observation alignment。DUET/SAME 通过大量 VLN 数据、拓扑图、cross-modal policy learning 获得这些能力。

LLM 更适合担任**局部 reasoning / verification / explanation 模块**，而非主导航器。

### 为什么 SAME 与 NavGPT 在 NavNuances 上互补

二者拥有不同 evidence 接口：

| Skill | LLM/MLLM 可能优势 | 训练型 VLN 可能优势 |
|---|---|---|
| DC（方向变化） | 方向词语义、显式推理 | heading-action alignment |
| LR（地标识别） | open-vocabulary grounding | 训练数据内 landmark grounding |
| RR（区域识别） | 语义区域理解 | topo-map / route prior / trained room patterns |
| VM（垂直移动） | 语义 elevation | 3D action / graph elevation / trained path |
| NU（数量理解） | counting/ordinal | history tracking，普遍弱 |

### 旧 STOP 项目的重新定位

旧路线 pipeline 中缺少的关键环节：

```
candidate endpoint 是否真的满足 instruction goal？
final endpoint 与 candidate endpoint 的证据差距是否足够大？
这个 episode 属于哪个 skill failure？
当前 policy uncertainty 是否足以触发额外推理？
```

STOP 应从"主任务"降级为"状态验证框架的一个评测切片"。

## 问题建模

### 系统边界

不重训主导航器（以 SAME 为 backbone）。研究对象是 backbone 之外的 verification layer。

```
Backbone policy B: 给定 instruction I 与 observation/history H_t，输出 action distribution、trajectory、candidate endpoints。
State s_t: 当前 viewpoint、heading、elevation、history、topological context。
Candidate set C: final endpoint + trajectory visited endpoints + top-k ranker candidates。
Uncertainty U: action entropy、stop probability shape、OSR-SR gap proxy、loop/revisit、score margin、progress inconsistency。
Evidence E: trace evidence、visual evidence、room/object/landmark metadata、topology/elevation、dialog evidence、instruction clause evidence。
Verifier V: 输入 I, s_t/c, E，输出 candidate 是否满足 instruction / progress / target 的 score 或 margin。
Decision D: 只在 V(candidate) - V(final) 足够大且 harm risk 低时介入。
```

### 因果链条

```
Instruction skill type
    -> required evidence type
        -> backbone internal representation adequacy
            -> uncertainty / confusion
                -> verifier trigger
                    -> evidence-grounded state decision
                        -> recovery or harm
```

旧 STOP 项目只覆盖了链条中的 trace evidence → gate/ranker → recovery or harm。

### 关键变量

| 变量 | 类型 | 可能实现 |
|---|---|---|
| `skill_type` | categorical | DC / NU / LR / RR / VM / endpoint / dialog-goal |
| `policy_uncertainty` | continuous | entropy、top1-top2 margin、stop probability variance |
| `trace_risk` | continuous | loop、revisit、trajectory length、late stop、candidate distance proxy |
| `visual_semantic_score` | continuous | CLIP / object tag / room classifier / MLLM caption match |
| `instruction_clause_alignment` | continuous / structured | current clause match、completed clause count、entity overlap |
| `topological_consistency` | continuous | room transition、elevation transition、candidate neighborhood |
| `verifier_margin` | continuous | best non-final vs final evidence score |
| `decision` | binary / multiclass | keep final / switch endpoint / ask verifier / no action |

## 假设

### H1：不同 instruction skill 需要不同 evidence budget

DC 和 LR 更依赖语言语义与视觉 grounding；RR 和 VM 更依赖 embodied topology、region boundary、elevation transition；NU 更依赖 explicit history / counting memory。统一 verifier 很可能不足，skill-aware portfolio 更有希望。

### H2：uncertainty-triggered verifier 比固定调用 verifier 更稳

只有当 backbone entropy、progress inconsistency、stop ambiguity、candidate margin 等指标触发时调用 verifier，更符合 AdaNav / VLN-Copilot 的趋势，且能控制 harm。

### H3：endpoint recovery 需要 final-vs-candidate evidence gap，而不只是 candidate score

旧 ranker 容易把 non-final candidate 排到前面，但无法保证它比 final 更安全。下一阶段必须比较 `V(candidate) - V(final)`，并对 harm-sensitive 数据集设置保守阈值。

### H4：NavNuances 应超出 benchmark 角色，成为 route 的能力坐标系

NavNuances 五类能力可以变成 skill-aware evidence 的组织方式。已有的 SAME vs NavGPT 互补结果可直接作为设计依据。

### H5：旧 endpoint negative result 可以转化为论文级贡献

表述为 diagnostic / evidence insufficiency，而非包装成强方法，旧项目具有价值。说明现代 VLN agent 的 endpoint oracle gap 与 trace-only recoverability 之间存在明显断层。

## 验证设计

### Stage A：Ability Atlas

目标：把模型能力差异系统化。最小产物为 `ability_atlas.csv`（model × dataset × split × metric × skill_type）和能力解释文档。

核心问题：R2R aggregate 与 NavNuances skill score 的相关性如何？SAME 强项来自 RR/VM 还是更一般的 topology/3D 行为？NU 为什么所有模型都弱？

### Stage B：Evidence Audit

目标：为 verifier 准备统一证据 schema，字段包括 episode_id、scan_id、instruction、skill_type、candidate_viewpoint、is_final、backbone_score、stop_prob、entropy、trace_features、room/object/landmark/elevation proxy、instruction_entities、instruction_clause_id、oracle_success_label、recovery_harm_label。

### Stage C：Offline Verifier v0

目标：先不做在线 agent，只做 offline verifier。候选实现从低到高成本：rule verifier（entity overlap + room/object proxy + trace margin）→ CLIP verifier → LLM verifier → MLLM verifier。

### Stage D：Uncertainty-Triggered Reasoning

目标：让 verifier 只在高不确定性时工作。trigger 候选包括 action entropy 高、stop margin 低、top-k endpoint score 接近、loop/revisit、instruction clause mismatch、skill-specific anomaly。

### Stage E：Skill-Aware Evidence Portfolio

目标：让不同 skill 调用不同 evidence。DC 优先方向 verifier，LR 优先 landmark verifier（CLIP/MLLM），RR 优先 room-state verifier，VM 优先 elevation verifier，NU 先做 count-memory diagnostic。

## 当前判断

当前处于 **Stage 0 冻结与 Stage 1 启动之间**。旧 endpoint completion 项目已完成强 negative diagnostic，证明了 trace-only 的边界。新路线已具备充分动机和文献支撑。

下一步应从 Ability Atlas 开始，将"现象"变成结构化表，为后续 evidence schema 和 verifier 提供输入。

## 决策分析

### 方向取舍

| 方向 | 决策 | 理由 |
|---|---|---|
| 继续 trace-only STOP gate/ranker | 停止作为主线 | 4.7/4.8 已显示 recovery≈harm，继续扫参价值低 |
| 直接做 NavNuances router | 暂缓 | 需先搞清 evidence 与 skill 的对应关系 |
| 完整复现 NavGPT-2 | 不作为近期主线 | 工程成本高，聚合性能未超过 SAME |
| 纯 LLM agent | 不作为主线 | zero-shot reasoning 有价值，但 aggregate navigation 弱 |
| SAME backbone + verifier | 作为主线 | 已复现，有多任务数据，有工程可控性 |
| AdaNav/VLN-Copilot 式 adaptive reasoning | 作为方法思想 | 适合低资源与低 harm 目标 |
| Long-horizon / continuous VLN | 作为远期连接 | 需要当前离散路线稳定后再推进 |

### 路线图

```
Stage 0: 冻结旧 endpoint trace-only negative diagnostic
Stage 1: Ability Atlas — 模型 × 数据集 × skill 的能力图谱
Stage 2: Evidence Audit — 构建 trace/visual/semantic/topology/dialog evidence schema
Stage 3: Offline Verifier — final-vs-candidate / state-vs-instruction 证据验证
Stage 4: Uncertainty Trigger — 只在高风险 step/episode 调用 verifier
Stage 5: Skill-Aware Evidence Portfolio — DC/LR/RR/VM/NU 分技能证据预算
Stage 6: 远期扩展 — long-horizon / continuous / step-aware dense progress
```

## 行动计划

### 六周计划

**Week 1：Ability Atlas**
- 建立 `ability_atlas.csv`，纳入 SAME、DUET、Mind、NavGPT、NavGPT-2、Human
- 合并 R2R/REVERIE/SOON/CVDN/NavNuances 结果
- 对 DC/NU/LR/RR/VM 写 evidence interpretation
- 停止条件：若数据源太散，先只纳入已有模型数据

**Week 2：Evidence Audit Schema**
- 从 SAME trace/endpoint candidate CSV 中抽取 final 与 top-k candidates
- 添加 instruction entity/clause parser 的最小版本
- 添加 room/object/landmark proxy 字段
- 添加 uncertainty features：entropy、stop margin、top-k margin、loop/revisit
- 停止条件：若 object/room proxy 暂时难接入，先用 instruction keyword + trace + topology

**Week 3：Offline Verifier v0**
- rule verifier：entity overlap、room keyword、candidate vs final evidence gap
- light learned verifier：LogReg/LightGBM，按资源选择
- dev 上选 threshold，val_seen 检查外推，暂不碰 val_unseen
- 最低继续线：recovered/harmed margin 有清晰差异

**Week 4：Uncertainty Trigger**
- 设计 trigger：entropy/stop margin/trace risk/verifier margin
- 对比 fixed verifier vs uncertainty-triggered verifier
- 记录 compute budget：episode 级调用比例
- 最低继续线：相近 dSR 下，trigger 版本 harm 更低或调用比例显著降低

**Week 5：Skill-Aware Portfolio**
- 给 instruction 生成 pseudo skill type
- DC/LR 优先尝试 LLM/CLIP evidence
- RR/VM 优先尝试 topology/room/elevation evidence
- NU 先做 count-memory diagnostic，不强求提升

**Week 6：Frozen val_unseen 一次性报告**
- 前提：dev/val_seen 达到继续线
- 冻结 verifier、trigger、threshold、dataset-specific rule
- 只跑一次 val_unseen，生成 recovery/harm、dSR/dSPL、dataset-wise、skill-wise 结果

### 每两周复盘问题

1. 新实验是否增加了 evidence separability？
2. recovery/harm 是否比旧 trace-only baseline 有明显改善？
3. 是否出现 dataset-specific harm，特别是 SOON/CVDN？
4. LLM/MLLM 是否只在明确 skill/uncertainty 下被调用？
5. 当前工作是否仍能写进 global/local/stage map，而非临时散点？
