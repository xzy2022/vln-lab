## 开发patch

正在开发的patch放在`patches/*/experimental`,等到确认稳健了,我会自己移动到`patches/*/base`

对 SAME 实验，experimental patch 不再靠手动改 `third_party/SAME` 工作树来隐式生效。需要参与某次实验时，通过运行器 CLI 显式声明:

```bash
python scripts/experiments/run_same.py \
  --config configs/same/val_unseen_r2r_reverie_cvdn_soon.yaml \
  --experimental-patch patches/same/experimental/0005-new-feature.patch
```

可以重复传入多个 `--experimental-patch`。

如果 patch 方式明显不适合，请直接说明理由，我会自己考虑 fork 子项目；不要死硬地只用 patch，也不要擅自直接改 `third_party/` 工作树。

## 移动patch

当patch经过检验后,我会主动发出移动patch的指令.这种情况下,把要求的patch从`experimental`移动到`base`,并更新`docs/patches`下面对应的文档.

注意: 移动 patch 不是简单 `mv` 文件。以前出现过的问题是,把 experimental patch 移到 base 后又用普通文本清理方式处理了 `.patch` 文件,导致 hunk 里的空白上下文行被改坏。`.patch` 文件内部每一行都有语义: 空格开头是上下文行,`+` 开头是新增行,`-` 开头是删除行。空白上下文行也必须保留前导空格,不能被 `s/[ \t]+$//` 这类全局尾随空白清理误删。否则文件看起来更干净,但 `git apply` 会报“补丁损坏”或“补丁未应用”。

移动 patch 时请按这个流程做:

1. 先确认目标补丁在当前 `experimental` 位置能应用,再移动到 `base`。
2. 不要直接根据脏的 `third_party/` 工作树手写 patch。优先从子项目干净 `HEAD` 建一个临时树,按顺序应用现有 `base/*.patch`,再应用或重生成要移动的 patch。
3. 如果需要重生成 patch,应在临时树中完成修改后用 `git diff -- <changed-files>` 生成,而不是手动编辑 hunk 行号、index 行或空白上下文。
4. 对 `.patch` 文件不要运行普通的全局尾随空白清理。`git diff --check` 可以检查文档、CSV、源码等普通文件,但不能作为 patch 文件是否正确的唯一依据。
5. 移动完成后必须在干净临时树里验证所有 base patch 可按文件名顺序应用。

推荐校验命令:

```bash
tmp=$(mktemp -d)
mkdir -p "$tmp/src"
git -C third_party/SAME archive HEAD | tar -x -C "$tmp/src"
for patch in patches/same/base/*.patch; do
  git -C "$tmp/src" apply --check "$PWD/$patch"
  git -C "$tmp/src" apply "$PWD/$patch"
done
rm -rf "$tmp"
```

只有上面的顺序 apply 检查通过后,再提交 `patches/*/base` 和 `docs/patches` 的变更。若检查失败,优先重生成对应 patch,不要靠手动改空白或改行号硬修。
